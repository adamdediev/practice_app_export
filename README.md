# АРМ руководителя производственной практики

Прототип программного модуля для учёта и оценки результатов прохождения
производственной практики студентами (ПМ.01, специальность 09.02.07).

---

## 1. Структура проекта

```
practice_app/
├── practice_record.py              # Бизнес-логика (класс PracticeRecord + исключение)
├── app.py                          # Графический интерфейс (tkinter)
├── tests/
│   └── test_practice_record.py     # Модульные тесты (unittest)
├── students.json                   # Тестовые данные (7 студентов, погран. случаи)
├── students.csv                    # Те же данные в CSV
├── uml_diagram.svg / .png          # UML-диаграмма классов (Модуль 1)
├── flowchart_determine_status.svg / .png   # Блок-схема алгоритма (Модуль 1)
└── README.md
```

## 2. Запуск

Требуется Python 3.10+ (tkinter входит в стандартную поставку).

```bash
# Запуск приложения с графическим интерфейсом
python app.py

# Запуск всех юнит-тестов
python -m unittest discover -s tests -v
```

## 3. Реализация требований ТЗ

| Требование ТЗ | Где реализовано |
|---|---|
| Регистрация ФИО и группы | `app.py` — поля ввода; `PracticeRecord.set_fio/set_group` |
| Три оценки за этапы практики | выпадающие списки `cb_diary/cb_task/cb_report` |
| Средний балл (округл. до 0.1) | `PracticeRecord.calculate_average()` |
| Статус «Допущен / Не допущен» | `PracticeRecord.determine_status()` |
| Правило: одна «2» → «Не допущен» | цикл поиска двойки в `determine_status()` |
| Запрет пустых ФИО/группы | валидация в сеттерах (`ValueError`) |
| Запрет текста/спецсимволов в оценках | `__validate_grade()` через `int()` |
| Диапазон оценок 2..5 | `ArgumentOutOfRangeException` |
| Обработка исключений try-catch | `app.on_calculate()` — блок `try/except` |
| Уведомление об ошибке | `messagebox.showerror(...)` |
| Цветовая индикация | зелёный фон для допущенных, красный — для нет |

## 5. Юнит-тесты (8 тестов, все проходят)

Обязательные по ТЗ:
- **Тест 1** `test_admission_success` — допуск при оценках 4, 4, 5.
- **Тест 2** `test_not_admitted_due_to_two` — недопуск при 5, 5, 2.
- **Тест 3** `test_grade_out_of_range_raises` — `ArgumentOutOfRangeException` при оценке 6.

Дополнительные: округление среднего, все пятёрки, текст вместо оценки,
пустое ФИО, проверка инкапсуляции (геттер возвращает копию списка).

## 6. Материалы для защиты (Модуль 3)

### 6.1. Почему логика расчёта отделена от интерфейса (SOLID)

Применён принцип **единственной ответственности (Single Responsibility, SRP)**:

- `PracticeRecord` отвечает **только** за данные, валидацию и расчёты;
- `PracticeApp` отвечает **только** за ввод/вывод (представление).

Преимущества такого разделения:
1. **Тестируемость** — бизнес-логику можно покрыть юнит-тестами без запуска GUI
   (именно поэтому тесты вообще возможны).
2. **Заменяемость интерфейса** — можно заменить tkinter на веб- или консольный
   интерфейс, не трогая ни строчки логики (близко к **принципу инверсии
   зависимостей, DIP**: UI зависит от абстракции «посчитать», а не наоборот).
3. **Сопровождаемость** — изменение бизнес-правила (например, порога допуска)
   не затрагивает код интерфейса.

### 6.2. Как работает алгоритм поиска «двоек»

Метод `determine_status()` перебирает массив из трёх оценок в цикле.
Как только встречается оценка, равная 2, метод **немедленно возвращает**
«Не допущен» (ранний выход — не нужно проверять остальные). Если цикл
завершился, не найдя ни одной двойки, возвращается «Допущен к аттестации».

Ключевой момент: проверка на двойку **не зависит от среднего балла**.
Студент Петров (5, 5, 2) имеет высокий средний 4.0, но всё равно не допущен —
это и демонстрирует приоритет бизнес-правила над арифметикой.

```python
for grade in self.__grades:
    if grade == 2:
        return "Не допущен"
return "Допущен к аттестации"
```

### 6.3. Инкапсуляция

Все поля приватны (`__fio`, `__group`, `__grades`). Доступ — только через
геттеры/сеттеры. Сеттеры выполняют валидацию, поэтому объект `PracticeRecord`
**невозможно** создать в некорректном состоянии. Геттер `get_grades()`
возвращает копию списка, чтобы внешний код не мог изменить внутренние данные
в обход проверок.

### 6.4. Надёжность

Обработчик кнопки `on_calculate()` обёрнут в `try/except` с раздельной
обработкой `ArgumentOutOfRangeException` (выход за диапазон), `ValueError`
(пустые поля/текст) и общего `Exception` (страховка). При любой ошибке
пользователь видит понятное сообщение, а приложение продолжает работу.
